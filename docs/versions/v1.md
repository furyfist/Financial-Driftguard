## Project Goal and the Meaning of "V1"

Financial DriftGuard is an open-source Python library for monitoring financial ML models in production. The core differentiator over horizontal tools (Arize, WhyLabs, Evidently) is a **regime classifier** that distinguishes genuine market regime changes (rate hikes, recessions, black swan events) from actual model decay — a distinction that has material consequences in credit risk operations. Retraining a credit model during a recession on recession data produces a model useless post-recovery. Standard tools cannot make this distinction; DriftGuard is designed to.

**V1 scope** was defined as: a working, end-to-end system demonstrating the core thesis on real data. It includes the Python drift detection library, a rule-based regime tagger, a FastAPI backend with persistence, a React dashboard, and a validated demo on the Lending Club dataset. V1 is explicitly **not** production-hardened — no auth layer, SQLite-only storage, rule-based (not ML) regime classification, no webhook notifications. These are V2/V3 concerns.

The primary validation case is a LightGBM credit default model trained pre-2016 and monitored through the 2017–2018 Federal Reserve hiking cycle.

## Delivered Components

### 1. Drift Detection Library (`driftguard/`)

#### Data Contract — `core/drift_result.py`

The foundational type layer. Defined first so the entire codebase agrees on what a drift result is.

- `DriftSeverity` — enum: `none`, `low`, `medium`, `high`, `critical`
- `FeatureDriftResult` — per-feature, per-detector result. Fields: `feature_name`, `detector`, `score`, `threshold`, `severity`, `p_value`, `details`
- `DriftResult` — top-level result. Fields: `model_id`, `checked_at`, `feature_results`, `overall_severity`, `regime`, `notes`
- Computed properties: `is_drifted`, `drifted_features`, `drift_score` (normalised 0–1 mean across all feature scores)

#### Abstract Base — `detectors/base.py`

`BaseDetector` defines the interface all detectors implement:
- Public method: `detect(feature_name, baseline, current) → FeatureDriftResult`
- Abstract method: `compute(baseline, current) → tuple[float, dict]`
- Internal: `_score_to_severity()` — maps raw score to `DriftSeverity` using class-level thresholds (`THRESHOLD_LOW`, `THRESHOLD_MEDIUM`, `THRESHOLD_HIGH`)
- Input validation and numpy casting handled in `detect()` — subclasses only implement math

#### PSI Detector — `detectors/psi.py`

Population Stability Index — the industry standard in credit model monitoring. Banks use this metric internally; its inclusion gives the library regulatory credibility.

Key implementation decisions:
- **Bin edges defined on baseline data only** — current data is bucketed using baseline boundaries. This is intentional and correct: you are measuring how much the current population has shifted relative to the reference
- **`np.percentile` binning** — percentile-based bins (not equal-width) handle skewed financial distributions better
- **`np.unique()` on bin edges** — prevents duplicate bins on low-variance features (e.g. binary flags like `pub_rec_bankruptcies`)
- **Epsilon smoothing (`1e-6`)** — prevents `log(0)` on empty bins
- Thresholds: `LOW=0.10`, `MEDIUM=0.20`, `HIGH=0.25` — these match industry convention for credit monitoring

#### KS Detector — `detectors/ks_test.py`

Two-sample Kolmogorov-Smirnov test via `scipy.stats.ks_2samp`. Catches distributional shape changes that PSI misses. Particularly effective on skewed financial features (income, loan amount).

- Score = KS statistic (maximum CDF distance), range [0, 1]
- `p_value` surfaced in `details` and as `FeatureDriftResult.p_value`
- `significant` flag (`bool`) in details — `True` when `p_value < 0.05`
- **Critical fix applied:** `scipy` returns `np.bool_` not Python `bool` — explicit `bool()` cast added after test failure caught this

#### JS Divergence Detector — `detectors/js_divergence.py`

Jensen-Shannon divergence via `scipy.spatial.distance.jensenshannon`. Symmetric and bounded [0, 1] — better than raw KL divergence when distributions have non-overlapping support, which is common after regime shifts.

- **Shared bin edges across both arrays** — unlike PSI, JS uses `np.linspace(combined_min, combined_max, n_bins+1)` for fair comparison
- Returns JS distance (square root of JS divergence), not raw divergence
- Epsilon smoothing on histograms before passing to `jensenshannon`

#### Data Ingestion — `core/snapshot.py`

`DataSnapshot` wraps a window of feature data at a point in time.

- `from_dataframe(df, label)` classmethod — selects only numeric columns, drops NaNs per feature, stores as numpy arrays
- `get(feature_name)` — raises `KeyError` with descriptive message if feature missing
- `captured_at` — timezone-aware UTC datetime (`datetime.now(timezone.utc)`)

#### Public SDK — `core/monitor.py`

`Monitor` is the single entry point users interact with.

```python
monitor = Monitor(model_id="lending_club_v1")
result  = monitor.check(baseline_snap, current_snap, macro=macro_snapshot)
```

- Accepts optional `detectors` list (defaults to all three: PSI, KS, JS)
- Accepts optional `features` list (defaults to all shared features between snapshots)
- `_resolve_features()` — computes intersection of baseline and current feature sets; raises `ValueError` if requested features are missing
- `_aggregate_severity()` — returns worst severity across all feature results
- Regime tagging is **lazy** — `RegimeTagger` is only imported and called if `macro` is not `None` (avoids circular import, keeps the path optional)
- **Deprecation fix:** `datetime.utcnow()` replaced with `datetime.now(timezone.utc)` throughout

#### Public API Surface — `driftguard/__init__.py`

Exposes: `Monitor`, `DataSnapshot`, `DriftResult`, `DriftSeverity`, `FeatureDriftResult`, `PSIDetector`, `KSDetector`, `JSDivergenceDetector`, `RegimeTagger`, `Regime`, `RegimeAssessment`, `MacroSignalFetcher`, `MacroSnapshot`

### 2. Regime Layer (`driftguard/regime/`)

The core intellectual contribution of the project. Nothing comparable exists in horizontal monitoring tools.

#### Macro Signal Ingestion — `regime/macro_signals.py`

`MacroSnapshot` — point-in-time dataclass holding:
- `vix` — CBOE Volatility Index (Yahoo Finance via `yfinance`)
- `credit_spread` — BAA minus 10Y Treasury spread (FRED series `BAA10Y`)
- `fed_funds_rate` — Federal Funds Rate (FRED series `FEDFUNDS`)
- `yield_curve` — 10Y minus 2Y Treasury spread (FRED series `T10Y2Y`)
- `unemployment_rate` — US unemployment rate (FRED series `UNRATE`)

`MacroSignalFetcher` — fetches live or historical macro data. Requires a free FRED API key (`https://fred.stlouisfed.org/docs/api/api_key.html`). **Degrades gracefully** — if key is absent or network fails, each signal returns `None` and a warning is logged. VIX is fetched independently via Yahoo Finance and does not require a FRED key.

`is_complete()` — returns `True` only if all five signals are non-None. Partial snapshots fall through to `_partial_assessment()` in the tagger.

#### Regime Tagger — `regime/tagger.py`

`Regime` enum: `stable`, `rate_shock`, `credit_stress`, `recession`, `black_swan`, `unknown`

`RegimeAssessment` dataclass: `regime`, `confidence` (0–1), `signals_fired` (list of threshold names), `recommendation` (plain-English string)

**Classification logic** (rule-based, V1):

Threshold constants:
```python
_VIX_STRESS      = 25.0
_VIX_CRISIS      = 40.0
_SPREAD_STRESS   = 1.50   # percentage points
_SPREAD_CRISIS   = 3.00
_UNEMP_RISING    = 6.0    # percent
_YIELD_INVERTED  = 0.0    # 10Y-2Y < 0 = inversion
```

Signal evaluation fires named flags (`vix_crisis`, `vix_stress`, `credit_spread_crisis`, `credit_spread_stress`, `yield_curve_inverted`, `unemployment_elevated`, `credit_conditions_tightening`). Classification then maps flag combinations to regimes in priority order: `black_swan` → `recession` → `rate_shock` → `credit_stress` → `stable`.

**Confidence scoring:** `min(len(signals_fired) / 6, 1.0)` with a +0.15 boost if VIX is in stress territory (VIX is the most reliable real-time signal).

**Recommendation logic** crosses regime with drift severity — the most important output. Key cases:
- Stable + drifted → "likely model decay, investigate pipeline"
- Rate shock / credit stress + drifted → "macro-driven, don't retrain, wait for stabilisation"
- Recession + drifted → "adjust thresholds, champion-challenger"
- Black swan → "freeze automated decisions, human review required"

**Validated outputs:**
- COVID macro snapshot (VIX=57, spread=3.8, unemployment=14.7) → `black_swan` ✓
- 2017–2018 Fed hiking cycle (VIX=17, spread=1.6, fed_funds=1.5) → `credit_stress` ✓

### 3. Backend API (`driftguard/api/`)

#### Database — `store/database.py`

Three SQLModel tables backed by SQLite (`driftguard.db` at repo root):

- `ModelRecord` — `model_id` (unique, indexed), `description`, `created_at`
- `DriftRun` — `model_id` (FK), `checked_at`, `overall_severity`, `drift_score`, `regime`, `notes`, `feature_results_json` (serialised JSON blob of all feature results)
- `AlertRecord` — `model_id`, `drift_run_id` (FK), `severity`, `message`, `acknowledged`, `created_at`

`create_db()` called on startup via FastAPI lifespan. `get_session()` is a dependency-injected generator.

#### FastAPI App — `api/main.py`

- CORS configured for `http://localhost:5173` (Vite dev server)
- Lifespan handler: `create_db()` + `start_scheduler()` on startup, `stop_scheduler()` on shutdown
- Routes registered: `/models`, `/drift`, `/alerts`
- `GET /health` returns `{"status": "ok", "version": "0.1.0"}`
- Interactive docs at `http://localhost:8000/docs`

#### Routes

`/models/` — `POST` (register), `GET` (list), `GET /{id}`, `DELETE /{id}`

`/drift/` —
- `GET /{id}/history?limit=50` — ordered by `checked_at` descending
- `GET /{id}/latest` — single most recent run
- `GET /{id}/features/{run_id}` — deserialised JSON from `feature_results_json`
- `POST /{id}/run` — accepts `records` (list of row dicts) + `set_as_baseline` bool. If baseline, calls `register_baseline()`. Otherwise calls `run_drift_check()` and returns summary

`/alerts/` — `GET /?unacknowledged_only=bool`, `GET /{model_id}`, `POST /acknowledge`

#### Scheduler — `scheduler/jobs.py`

`APScheduler BackgroundScheduler` wired into FastAPI lifespan. In-memory baseline store (`_baselines: dict[str, DataSnapshot]`) keyed by `model_id`.

`run_drift_check(model_id, current)` — runs `Monitor.check()`, serialises feature results to JSON, persists `DriftRun` to SQLite, creates `AlertRecord` if severity is `high` or `critical`.

> **Limitation:** Baselines are in-memory only. Server restart loses all registered baselines. V2 will persist baselines to the store.

### 4. React Dashboard (`dashboard/`)

**Stack:** Vite + React 18 + TypeScript + Tailwind CSS v3 + Recharts + Axios + React Router

**Design system:**
- Font pair: `Bricolage Grotesque` (display/body) + `DM Mono` (numbers, scores, metadata) — both Google Fonts
- Background: `#F7F6F3` (warm off-white)
- Accent: `#D4450C` (burnt orange — alert colour + brand)
- Stable: `#1A6B3C`, Warning: `#B45309`, Critical: `#C0200F`
- All tokens defined as Tailwind custom colours in `tailwind.config.js`

**Known setup issue encountered:** `npx tailwindcss init -p` failed to generate `postcss.config.js` because the directory was not empty. File must be created manually. Tailwind v4 was also accidentally installed initially — must pin to `tailwindcss@3`.

#### Components

`RegimeBadge` — maps `Regime` enum to pill with coloured dot. Dot pulses (CSS `pulse-dot` animation) for any non-stable regime. Renders `unknown` gracefully when regime is null.

`SeverityBar` — horizontal progress bar where width maps to severity enum (`none=8%` → `critical=100%`). Colour shifts from green through amber to red. Score displayed in `DM Mono` to the right.

`ModelHealthCard` — fetches latest drift run on mount via `driftApi.latest()`. Shows model name, description, `RegimeBadge`, `SeverityBar`, severity label, and last-checked date. Left border colour encodes severity. Clickable, navigates to `ModelDetail`.

`AlertFeed` — renders unacknowledged alerts with coloured dots and Ack buttons. Critical alerts pulse. Calls `alertsApi.acknowledge()` on click and triggers parent refresh.

#### Pages

`Overview` — fetches all models and unacknowledged alerts in parallel. Renders `ModelHealthCard` grid + `AlertFeed`. Unacknowledged count shown in header pill.

`ModelDetail` — fetches drift history and feature results for latest run. Shows three summary stat cards (drift score, severity, regime), recommendation banner in accent colour, `LineChart` of drift score over time with reference line at `0.25`, feature PSI table sorted by score descending.

#### API Client — `api/client.ts`

Typed axios wrapper. Three namespaced exports: `modelsApi`, `driftApi`, `alertsApi`. All methods return typed promises.

### 5. Demo — Lending Club (`demo/lending_club.py`)

#### Training (Kaggle notebook, not in repo)

- Dataset: Lending Club accepted loans 2007–2018 Q4 (2.26M rows, 151 columns)
- Filtered to 2013+ (earlier loans have poor data quality)
- Target: binary default label from `loan_status` (`Charged Off`, `Default`, and related statuses)
- 17 origination-time features selected — no post-origination features to prevent leakage
- **Time-based split (not random):** Train 2013–2015 (791k rows), Baseline 2016 (434k rows), Live 2017–2018 (938k rows)
- Algorithm: LightGBM with `scale_pos_weight = neg/pos = 4.74` for class imbalance
- Validation AUC: **0.6919** (industry-realistic for credit default without leakage)
- Live AUC: 0.6969 (marginally higher — 2017–2018 loans had more maturation time)

Artifacts saved to `demo/data/` (5,000-row samples for baseline and live):
- `lending_club_model.pkl`
- `baseline_snapshot.parquet`
- `live_snapshot.parquet`
- `feature_columns.json`

> **Compatibility warning:** Model pickled with sklearn 1.0.2 (Kaggle), loaded with sklearn 1.8.0 locally. Produces `InconsistentVersionWarning` on load. Results are unaffected but this should be resolved in V2 by re-training locally or pinning sklearn.

#### Demo Script Output

Recommendation   : Drift consistent with macro regime shift. Model is correctly
                   uncertain. Monitor closely but avoid retraining on regime data

**Interpretation:** `int_rate` drift leading (PSI 0.1013, crossing the 0.10 threshold) correctly reflects the Federal Reserve hiking cycle beginning in 2016–2018. Rates on new loans shifted first; the real economy had not yet felt the impact. DriftGuard correctly tagged this as `credit_stress` (macro-driven) rather than model decay. ## Implementation Decisions and Patterns

| Decision | Rationale |
|---|---|
| PSI bin edges defined on baseline only | Correct statistical practice — measures deviation of current from reference |
| Lazy regime import in `Monitor.check()` | Avoids circular import between `core` and `regime` packages |
| `feature_results_json` blob in `DriftRun` | Avoids a separate join table for V1; acceptable at this scale |
| In-memory baseline store in scheduler | Simplest V1 approach; acknowledged limitation, persistence planned for V2 |
| Tailwind v3 (not v4) | v4 dropped `tailwind.config.js` support; v3 required for custom token system |
| LightGBM over XGBoost for demo | Handles NaNs natively (Lending Club has many), faster, better-calibrated probabilities |
| Time-based split on `issue_d` | Random splits leak future data into training on time-series financial data |
| `scale_pos_weight` over `class_weight` | Better imbalance handling in LightGBM specifically |
| Pre-release tag | No auth layer, SQLite-only, rule-based tagger — honest about production readiness |

## Known Gaps and Intentional Omissions

| Gap | Notes |
|---|---|
| Regime shows `unknown` in dashboard | `seed_api()` does not pass a `MacroSnapshot` to the API run. Demo script shows correct `credit_stress` when called directly. Fix in V2: macro context endpoint |
| Baseline lost on server restart | In-memory only. Needs persistence to SQLite in V2 |
| No API authentication | Any client can register models or trigger drift checks. Acceptable for local use, not for deployment |
| SQLite only | Not suitable for concurrent multi-user production use. PostgreSQL migration planned for V3 |
| Rule-based regime tagger | Confidence scoring is heuristic. V2 replaces with a trained classifier on NBER recession dates and historical VIX regimes |
| `sklearn` version mismatch on model load | LGBMClassifier pickled with sklearn 1.0.2 (Kaggle default), loaded with 1.8.0 locally. Warning is cosmetic, results unaffected |
| `adapters/` folder is a stub | `sklearn_adapter.py` and `torch_adapter.py` exist as empty placeholders. Model-specific wrapping not implemented in V1 |
| `drifted_features` count misleading | Counts detector hits, not unique features. A feature appearing in PSI + KS + JS counts three times. Display bug, not a logic bug |
| No webhook notifications | Discord/Slack/Telegram planned for V2 |
| FRED API key required for full regime | Without a key, only VIX is available. Tagger falls back to partial assessment with 0.3 max confidence |
| `demo/data/` not in git | Must be re-generated from Kaggle notebook. No reproducibility script committed yet |

## How to Run From Scratch

```bash
# 1. clone and set up environment
git clone https://github.com/<username>/financial-driftguard
cd financial-driftguard
python -m venv venv
venv\Scripts\activate        # Windows
pip install -r requirements.txt
pip install -e .

# 2. verify library works
python scripts/sanity_check.py

# 3. run tests
pytest tests/test_detectors.py -v

# 4. place Kaggle demo artifacts in demo/data/ (see training notebook)

# 5. start backend
uvicorn driftguard.api.main:app --reload

# 6. register model and seed data (backend must be running)
python demo/lending_club.py

# 7. start dashboard (separate terminal)
cd dashboard
npm install
npm run dev
# open http://localhost:5173
```
